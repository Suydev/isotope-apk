1. PURPOSE
This SQL file is a comprehensive cumulative patch (v4 through v12) for the IsotopeAI platform. It is designed to sync the database schema with the requirements of the frontend JS code. It handles table updates, missing table creation, RPC replacement, and storage security. It should be run on existing Supabase projects to upgrade them to the latest stable schema.

3. ROW LEVEL SECURITY (RLS)
- public.users: Enabled.
  - "users_read_all": SELECT USING (true).
  - "users_update_own": UPDATE USING (auth.uid() = id).
- public.groups: Enabled.
  - "groups_read_public": SELECT USING (is_public = true OR auth.uid() = owner_id).
  - "groups_member_read": SELECT USING (public._is_group_member(id, auth.uid())).
- public.group_members: Enabled.
  - "gm_read": SELECT USING (user_id = auth.uid() OR public._is_group_member(group_id, auth.uid())).
  - "gm_insert": INSERT WITH CHECK (auth.role() = 'authenticated').
- public.group_chat_messages: Enabled.
  - "gchat_read": SELECT USING (public._is_group_member(group_id, auth.uid())).
  - "gchat_insert": INSERT WITH CHECK (user_id = auth.uid() AND group_id IN (SELECT group_id FROM public.group_members WHERE user_id = auth.uid())).
- public.user_onboarding: Enabled.
  - "user_onboarding_select_own": SELECT USING (auth.uid() = user_id).
  - "user_onboarding_insert_own": INSERT WITH CHECK (auth.uid() = user_id).
  - "user_onboarding_update_own": UPDATE USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id).
- public.user_tours: Enabled.
  - "user_tours_own": ALL USING (auth.uid() = user_id).
- Storage (storage.objects):
  - "user-content: users manage own objects": ALL USING (bucket_id = 'user-content' AND auth.uid()::text = split_part(name, '/', 1)).
  - "notes: users manage own objects": ALL USING (bucket_id = 'notes' AND auth.uid()::text = split_part(name, '/', 1)).
  - "avatars: public read": SELECT USING (bucket_id = 'avatars').
  - "avatars: users manage own": ALL USING (bucket_id = 'avatars' AND auth.uid()::text = split_part(name, '/', 1)).

4. FUNCTIONS & TRIGGERS
- public._is_group_member(gid, uid): SQL, STABLE, SECURITY DEFINER. Purpose: Check membership without recursion.
- public.handle_new_user(): PLPGSQL, SECURITY DEFINER, search_path = public. Purpose: Seed all user-related tables on auth.users insert. Side effect: Inserts into multiple tables.
- public.sync_user_onboarding_from_profile(): PLPGSQL, SECURITY DEFINER, search_path = public. Purpose: Keep onboarding table in sync with profile_data JSON.
- public.set_user_tours_updated_at(): PLPGSQL. Trigger function to update timestamp.
- Trigger: on_auth_user_created ON auth.users (AFTER INSERT). Calls handle_new_user().
- Trigger: tr_sync_user_onboarding_from_profile ON public.user_profiles (AFTER INSERT OR UPDATE). Calls sync_user_onboarding_from_profile().
- Trigger: trg_user_tours_updated_at ON public.user_tours (BEFORE UPDATE). Calls set_user_tours_updated_at().

5. MIGRATIONS
This file assumes a basic Supabase schema might exist or be empty. It uses IF NOT EXISTS for everything. It adds columns to existing tables and creates many new satellite tables for features like stats, logs, and onboarding.

6. DATA SEEDING
- Seeds storage.buckets (avatars, user-content, notes).
- Backfills existing users' plan_type, billing_status, and satellite records (points, stats, profiles, presence).
- Backfills usernames from auth metadata or email.
- Originally seeded community_events (but v10 drops those).

7. IDEMPOTENCY
Highly idempotent. Uses:
- ALTER TABLE ... ADD COLUMN IF NOT EXISTS
- CREATE TABLE IF NOT EXISTS
- CREATE OR REPLACE FUNCTION
- DO $$ blocks with IF NOT EXISTS checks for constraints and policies.
- DROP POLICY IF EXISTS before CREATE POLICY.

8. FINDINGS
- BUG: Line 2026/2177 - The handle_new_user function is redefined multiple times in the file with slightly different logic (v4 vs v9). While idempotent, it shows inconsistent maintenance. Severity: LOW.
- SECURITY: Line 1353 - _is_group_member is SECURITY DEFINER but missing search_path in the v5 block, although later blocks might fix it. (Update: fixed in v8/v9). Severity: INFO.
- SECURITY: Line 2094 - sync_user_onboarding_from_profile uses SECURITY DEFINER with search_path=public. Correct.
- CODE SMELL: Massive duplication between blocks. The file is a concatenation of many patches. Severity: LOW.
- PERFORMANCE:GIN index on groups.fts is good for search.
- DATA INTEGRITY: Line 1134 - handle_new_user ON CONFLICT DO UPDATE resets plan_type to 'ranker' every time. If a user is manually upgraded to 'premium', a re-trigger might downgrade them. Severity: MEDIUM.
