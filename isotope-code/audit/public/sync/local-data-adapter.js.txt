PURPOSE
Provides a unified interface for reading and writing data to local storage (IndexedDB and LocalStorage). It abstracts the complexities of store management and synchronization events.

ROLE IN THE APPLICATION
- Subsystem: Persistence / Data Access
- Runtime stage: Application Lifecycle
- Active/legacy: Active

REFERENCES AND CONSUMERS
- Used by all functional modules to persist tasks, sessions, etc.
- Referenced by sync-size-optimizer.js.

INTERNAL STRUCTURE
- IndexedDB management (DB_NAME='isotope_main', VERSION=20).
- readCollection(name) / writeCollection(name, records).
- readAllLocalData() / writeAllLocalData(data).
- applyBackupToLocal(backupJson).

INPUTS
- IndexedDB (isotope_main).
- LocalStorage (isotope_*_v2).

OUTPUTS AND SIDE EFFECTS
- Database writes.
- LocalStorage updates.
- Dispatching 'isotope:sync_refresh' events.

DATA FLOW
1. App components call writeCollection.
2. Adapter updates IndexedDB and LocalStorage (for redundancy/legacy).
3. Adapter dispatches event to notify other components of the change.

SECURITY AND PRIVACY
- Stores all user data locally.
- No encryption implemented at this layer (standard for browser storage).

OFFLINE AND LOCAL-FIRST BEHAVIOUR
- The primary driver for local-first storage. Ensures data is available immediately without network.

PERFORMANCE
- Uses IndexedDB for bulk data to avoid blocking the main thread (unlike LocalStorage).

FINDINGS
- LD-001: Dual-writes to LocalStorage and IndexedDB for some collections. (Severity: INFO, Status: CONFIRMED)

POSITIVE OBSERVATIONS
- Clean abstraction of persistence.
- Robust schema upgrade logic.

FINAL VERDICT
Solid persistence layer for local-first apps.
